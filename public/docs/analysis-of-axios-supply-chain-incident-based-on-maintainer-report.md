# Axios npm 공급망 사고 분석 보고서

**참조**: [Axios GitHub Issue #10636 — Post Mortem: axios npm supply chain compromise](https://github.com/axios/axios/issues/10636)

---

## 개요

이번 사고는 Axios 코드베이스 취약점이 아니다. 리드 메인테이너를 겨냥한 표적형 사회공학 공격과 단말 침해를 통해 npm 배포 권한이 악용된 공급망 사고다. 공격자는 실존 기업인을 사칭하고, 정상적인 협업 흐름을 위장해 신뢰를 형성한 뒤 RAT를 설치했다. 이후 장악된 단말 또는 그 단말에서 사용 가능했던 게시 자격증명을 이용해 악성 버전을 공식 경로로 게시했다.

핵심 문제는 **개인 단말과 개인 계정에 종속된 게시 구조, 그리고 장기 자격증명 또는 재사용 가능한 게시 권한의 존재**에 있었다. 이번 사고는 계정 보호 수단의 유무보다, 배포 체인 자체가 사람의 단말 상태에 과도하게 의존하고 있었다는 점을 보여준다.

---

## 1. 침해 원인

공격의 출발점은 저장소가 아니라 사람과 단말이었다. 공격자는 약 2주에 걸쳐 사회공학 캠페인을 전개했고, 메인테이너의 PC에 RAT를 설치한 뒤 npm 게시에 활용 가능한 권한을 확보했다. 코드 자체에는 취약점이 없었다.

---

## 2. 공격 기법

단순 피싱이 아니다. 사칭 인물, 협업 도구, 화상회의, 소프트웨어 설치 유도가 결합된 다단계 공격이다.

- 실존 기업 창업자를 사칭하고 브랜드·인물 이미지를 정교하게 복제
- 실제처럼 보이는 Slack 워크스페이스 구성 (채널, 팀원 프로필, 메인테이너 계정 포함)
- MS Teams 화상회의로 초대해 업무 협업 흐름을 연출
- 회의 중 "시스템 구성 요소 업데이트" 명목으로 소프트웨어 설치 유도
- 피해자가 설치한 항목이 실제로는 RAT

개발자가 일상적으로 접하는 협업 흐름 자체를 공격 벡터로 활용한 점이 이 캠페인의 핵심 위험성이다.

---

## 3. 단말 침해와 게시 자격증명 악용

메인테이너 계정에는 2FA가 활성화되어 있었지만, 공개된 정보만으로 인증이 정확히 어떤 방식으로 우회되었는지는 확인되지 않았다. 보다 정확한 해석은, RAT 설치 이후 공격자가 해당 머신 자체를 제어하거나, 그 머신에서 사용 가능했던 게시 자격증명·세션·토큰 등을 악용했을 가능성이 높다는 것이다.

따라서 이번 사고를 "2FA 실패"로 요약하는 것은 부정확하다. 핵심은 **개인 단말에 종속된 게시 구조와, 단말 침해 이후 재사용 가능한 게시 권한이 존재했을 가능성**에 있다. 즉, 문제의 본질은 인증 수단 하나의 강도보다, 게시 체계 전체가 단말 신뢰에 과도하게 의존하고 있었다는 점이다.

---

## 4. 구조적 취약점

### 개인 단말 기반 배포

개인 계정과 개인 단말에서 직접 게시하는 구조가 이번 사고의 핵심 전제 조건이었다. 사고 이후 OIDC 기반 게시와 immutable release 체계를 도입하겠다고 밝힌 것 자체가, 배포 권한이 개인에게 지나치게 직접적으로 묶여 있었음을 보여준다.

### 탐지 자동화 부재

무단 게시를 실시간으로 감지하는 내부 자동화 체계가 없었다. 실제 발견은 커뮤니티 제보와 외부 탐지에 의존했다. 공급망 보호는 배포 예방만이 아니라 **비정상 릴리스 탐지**까지 포함해야 한다.

### 메인테이너 = 공급망 허브

고가치 오픈소스 프로젝트에서 배포 권한을 가진 메인테이너는 사실상 공급망의 핵심 진입점이다. 저장소 변조나 악성 PR 없이도, 개인 단말 하나의 침해만으로 수많은 다운스트림 환경에 악성 코드가 전파될 수 있다.

---

## 5. 조직 관점 시사점

- 신뢰된 외부 패키지라도 배포 주체가 침해되면 정상 유통 경로를 통해 악성 코드가 유입된다
- 공급망 리스크 평가는 CVE와 코드 취약점 점검만으로 충분하지 않다
- "누가, 어떤 환경에서, 어떤 방식으로 배포하는가" 자체가 보안 통제 대상이다
- 관리 범위를 **코드 저장소에서 배포 체인 전체**로 확장해야 한다

실질적으로 다뤄야 할 통제 항목:

| 영역 | 현재 위험 | 권고 방향 |
|------|----------|----------|
| 배포 환경 | 개인 단말 직접 게시 | CI/CD 기반 OIDC 배포, 개인 자격증명 제거 |
| 계정 보안 | 단말 의존적 게시 구조 | 엔드포인트 보안 병행, 장기 토큰·세션 제거 |
| 릴리스 무결성 | 사후 검증 없음 | provenance, Sigstore, immutable release 도입 |
| 탐지 | 외부 제보 의존 | 배포 이벤트 모니터링 자동화 |

---

## 결론

이번 Axios 사고는 라이브러리 취약점이 아니라, **배포 권한을 가진 사람을 겨냥한 공격**이다. 공격자는 개발자 생태계가 익숙하게 받아들이는 협업 도구와 업무 흐름을 위장해 신뢰를 쌓았고, 단말 장악 이후 공식 배포 경로 또는 그에 준하는 게시 권한을 그대로 이용했다.

방어 전략의 핵심은 세 가지다.

1. **배포 권한을 개인 단말에서 분리** — OIDC 기반 CI/CD 게시 체계
2. **엔드포인트 보안을 계정 보안과 동등하게 취급** — EDR, 장기 자격증명 제거
3. **비정상 릴리스 탐지 자동화** — 배포 이벤트 모니터링 내재화

유사한 장기 신뢰 형성형 사회공학 캠페인은 반복될 가능성이 높다. 오픈소스 의존도가 높은 조직일수록 메인테이너 침해 시나리오를 위협 모델에 명시적으로 포함해야 한다.
